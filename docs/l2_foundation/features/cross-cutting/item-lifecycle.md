# Library Item ライフサイクル契約

> Library item の「いつ追加され、 いつ消され、 何を消すのか」 を網羅した横断契約ドキュメント。 アイテム削除まわりを場当たりに 1 パターンずつ直してきた結果バグが続出 (ウィジェット削除でアイテムが孤立 / ページ削除で手動追加アイテムまで消える 等) した反省から、 全パターンを 1 表に集約し「正しい挙動」 を確定する。

## 用語と所有モデル

### 所有モデル

| 種別                  | 識別                           | 所有者               |
| --------------------- | ------------------------------ | -------------------- |
| **手動 item**         | `source_widget_id IS NULL`     | user (Library 全体)  |
| **監視自動登録 item** | `source_widget_id IS NOT NULL` | 当該監視ウィジェット |

`source_widget_id` / `source_entry_key` 列は migration 039 (PH-CF-100) で追加された back-link。 自動登録経路はこの 2 列を必ず埋める (= 片肺禁止)。

### DB 制約レイヤ (アプリ層 dedupe との関係)

| 対象                                                 | 制約                                                  | 結果                                                                                                                                                                               |
| ---------------------------------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `items.target`                                       | **UNIQUE 無し** (`001_initial.sql`)                   | 同一 target の二重登録は **アプリ層 dedupe (`find_by_target` / `find_by_source`) のみで防止**。 §D2 (bulk delete) / §C9 (import) / CLI / seed 経路では dedupe を経由せず行を作れる |
| `items.source_widget_id` / `source_entry_key`        | 部分 INDEX のみ (`039_items_source_back_link.sql:28`) | UNIQUE 無し → 同 widget の同 entry_key で複数行を作れる (発生しないのは `auto_register_*` 系が `find_by_source` で先に確認するからのみ)                                            |
| `widget_item_hides (widget_id, item_target)`         | PRIMARY KEY (`029_widget_item_hides.sql:19`)          | hide は idempotent (`INSERT OR IGNORE`)、 破綻しない                                                                                                                               |
| `watched_paths.path`                                 | UNIQUE だが **正規化なし** (`003_watched_paths.sql`)  | `C:\x` と `C:/x` は別行として両方登録可能。 現状 `ensureWatchedPath` は UNIQUE 違反を silent skip するため、 区切り混在で重複監視が成立しうる (Codex VERIFY 2026-05-25 §4 指摘)    |
| `workspace_widgets.config` の `item_id` / `item_ids` | TEXT (JSON)、 FK 無し                                 | items への dangling 参照は app 層の `cascade_remove_item_from_widgets` でのみ防止                                                                                                  |

「中途半端な孤立 item」 のほとんどは「app 層 dedupe / cascade を踏まない経路で行を作る or 残す」 ことで発生する。

### 本契約の scope 外

| 経路                                               | 理由                                                                                                                                                                                             |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `src-tauri/src/bin/arcagate_cli.rs`                | 開発者向け CLI。 本番アプリの起動経路には含まれない。 ただし backup / 復元用途で勝手に走らせると app 層 dedupe を回避するため、 user 教育で吸収する                                              |
| `src-tauri/examples/seed_dev.rs`                   | 開発 seed。 release ビルドに含めない                                                                                                                                                             |
| `script-folder` widget (`ScriptFolderWatchWidget`) | **items 行を作らない** widget (実体は watched_paths のみ作る `.sh` / `.bat` 監視)。 本契約の生成パターン C1-C9 には登場しない。 ただし widget 削除時の `watched_paths` 解除が漏れている (→ §D19) |

### 参照経路

item を参照する 5 系統:

1. `item_tags` (`sys-ws-<workspace_id>` / `sys-starred` / `sys-type-*` / user tag) — FK CASCADE
2. `workspace_widgets.config` JSON の `item_id` / `item_ids` キー — FK ではない (明示 strip 必要)
3. `widget_item_hides` (`widget_id`, `item_target` = scan entry key) — widget_id FK CASCADE
4. `launch_log` / `item_stats` / `confirmed_items` — FK CASCADE
5. `items.source_widget_id` 自身 (back-link) — `ON DELETE SET NULL` (migration 039)

## 確定済原則 (2026-05-25 user 承認)

1. **手動 item は自動 cascade で消えない**。 explicit user delete (Library 削除 / bulk delete) のみで消える。
2. **監視自動登録 item は所有ウィジェットの削除で削除**。 ただし他 workspace / 他 widget からも参照されているものは残す。
3. **workspace のページ / タブ削除では手動 item を消さない**。 配置 (widget config item_ids) を外すだけ。 消えるのは監視由来の自動登録 item のみ。
4. **Library から監視由来 item を直接削除** → 当該ウィジェットの除外リスト (`widget_item_hides`) に記録、 次の scan で復活させない。 復元は監視ウィジェット設定から (PH-CF-100 / PH-CF-500)。
5. **「画面に見えるのに編集すると壊れる中途半端な孤立 item」 を絶対に生まない**。 すべての削除経路で関連参照 (widget config item_ids / widget_item_hides / launch_log) を後始末する。
6. 上記で決めきれないパターン (再 scan で元ファイル消失 等) は §未確定パターン で選択肢 + 推奨案を提示。
