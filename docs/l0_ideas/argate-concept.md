# Argate 構想書

**PC上の全てを一箇所に集めて起動・管理するランチャー。**

## 1. 背景・目的

### 背景

現状、以下の課題がある。

- アプリの起動元がバラバラ（Steam、DMMゲーム、ブラウザゲーム、各種ランチャー等）で一元管理できていない
- 同一アプリの複数バージョン（Blender等）の切り替えが煩雑
- スクリプト実行（Claude Code起動等）に手数がかかる
- Windowsエクスプローラーの検索が遅く、代替ツールはモダンなUIでないか動作が重い
- PC上の作業ログが蓄積されていない

### 目的

「起動元・種別を問わず、あらゆるものを登録・整理・起動できる個人用コントロールパネル」を構築する。日常的に開くアプリとして、ランチャーとファイル管理を一体化する。詳細なアクティビティ記録はActivityWatchに委譲し、起動ログの最小限の記録のみ自前で持つ。ワークスペースによるビジュアルな操作をメインとしつつ、キーボード・CLIによるパワーユーザー向けの操作も両立する。

---

## 2. 制約

- **コスト：** 個人プロジェクトのためゼロコスト運用を基本とする
- **プラットフォーム：** Windows（x86-64）を優先、将来的なクロスプラットフォームを視野に入れた設計
- **技術スタック：** Tauri + Svelte + Rust
- **軽量性：** 常駐アプリとして起動コスト・メモリ使用量を最小化する
- **スコープ管理：** マイルストーン1を優先し、段階的に開発する

---

## 3. 機能要件

### マイルストーン1：ランチャー単体

「登録して起動できる」だけの最小構成。毎日使えるものを最速で作る。UIの中核は**コマンドパレット（起動特化）**。

**マイルストーン1のキラー体験：** 「ホットキーで呼び出し → 名前を打つ → Enterで即起動」がストレスゼロで動くこと。

**登録できる対象**
- 任意のコマンド・実行ファイル（.exe、.lnk）・URL・フォルダショートカットを登録可能
- ラベル・アイコン・カテゴリを自由に設定
- 例：Blender複数バージョン、PowerShellスクリプト、Claude Code、ブラウザゲーム等

**登録UX**
- exeのドラッグ＆ドロップで即登録
- アイコンの自動取得（exe→アイコン抽出、URL→favicon取得）
- 右クリックメニューによるアイテムの編集・削除・移動

**分類・整理**
- カテゴリ/グループによる整理
- タグ付け（マイルストーン2以降の拡張を見越したスキーマ設計）

**起動**
- ワンクリック/ホットキーによる起動
- スクリプト実行時のターミナル出力表示オプション（最低限：実行後にログを別ウィンドウで開く程度。PTY統合・エンコーディング対応などへの作り込みは避ける）

**起動・常駐**
- Windows起動時の自動起動
- システムトレイへの常駐
- ホットキーでどこからでも即呼び出し
- 常駐時のメモリ使用量を最優先で最小化

**起動ログ（最小限）**
- 起動回数・最終起動日時をSQLiteに記録
- 「最近起動したもの」「よく起動するもの」を表示

**オンボーディング**
- 初回起動時のシンプルなセットアップウィザード

**バックアップ・エクスポート**
- SQLiteデータ一式のエクスポート/インポート

---

### マイルストーン2：ワークスペース・クイックアクセス・MCP

マイルストーン1が安定してから着手。「使いやすく・AIと繋がる」段階。3本立てのため2a/2b/2cに分割して進める。

#### マイルストーン2a：コマンドパレットの拡張

マイルストーン1のコマンドパレット（起動特化）を拡張し、内蔵コマンド・CLI・名前空間を追加する。

**CLI対応**
- CLIからもコマンドパレットを操作可能（例：`argate run blender`）

**内蔵コマンドの追加**
- クリップボード履歴（コピー履歴の検索・再利用）
- スニペット（キーワードで定型文を展開、例：`@adr`→住所）
- 電卓（計算式を入力して即結果取得）

**カテゴリ名前空間（プレフィックス）**
- M2a時点では「カテゴリ/グループ」をネームスペースとして扱う
- カテゴリにショートプレフィックスを設定可能（例：Gamesカテゴリ → `gm`）
- `gm` と打つとそのカテゴリのアイテム一覧、`gm sakura` で絞り込み起動
- ページ概念はマイルストーン2bで導入。2b以降はページがネームスペースになる

**パス変更の自動追跡**
- ファイルシステム監視によるパス変更の自動追跡（NTFSファイルIDベース）
- 追跡不能になった場合は警告を表示

---

#### マイルストーン2b：ワークスペース

**実装方針：** 初期は内蔵ウィジェットのみで動く固定構成から始める。プラグインとして拡張できるインターフェースだけを先に定義し、サードパーティ対応は後続マイルストーンで解放する。

**ワークスペース・ウィジェット構成**
- ユーザーが任意にページを追加・定義し、好きなウィジェットを自由に配置
- 初期内蔵ウィジェット：よく使うもの・最近使ったもの・プロジェクト一覧・監視フォルダ
- ウィジェットの表示/非表示・レイアウトをページごとに設定可能
- 初回起動時にワークスペースのテンプレートを選択可能（例：ゲーム向け・開発向け・汎用）
- ページ導入以降は、ページがコマンドパレットのネームスペースになる（2aのカテゴリ名前空間を引き継ぐ）
- 拡張ポイント（プラグインインターフェース）は定義しておき、サードパーティ製ウィジェットはマイルストーン4以降で対応

**テーマ**
- テーマの切り替えによるビジュアルカスタマイズ
- カラースキーム（ダーク・ライト・カスタムカラー）
- UIの角丸・ボタンの丸み等のシェイプ設定
- テーマのインポート・エクスポート（共有可能）

**プロジェクト管理**
- プロジェクトフォルダへのクイックアクセス
- Gitステータス表示（ブランチ名・未コミット変更・最終コミット日時）

**監視フォルダ（自動登録）**
- フォルダを「監視対象」として登録し、配下に追加されたアイテムを自動でランチャーに登録
- フォルダごとに「追加時のカテゴリ」「アイテム種別」を設定可能
- 例：Projects/ → 新フォルダ＝プロジェクト、Tools/ → 新実行ファイル＝ツール

---

#### マイルストーン2c：MCP

**MCPサーバー（デフォルト提供）**
- このプロダクト自体がMCPサーバーを内蔵し、Claude Codeからの操作をデフォルトで受け付ける
- 2c時点の対応操作はアイテムの登録・起動・検索に限定する
  - 例：`blender`起動、`VSCode`でプロジェクトを開く、登録済みアイテムの検索
- ワークスペース操作・ファイル操作などの権限はマイルストーン4以降で段階解放する
- 初回セットアップ含めClaude Codeへの自然言語指示で全設定を完結させることを目指す
- プラグインAPIもMCP経由で操作可能にし、サードパーティ製プラグインも含めAIから全操作できる設計とする

---

### マイルストーン3：コンテキストメニュー統合

マイルストーン2aが安定してから着手。実装難度に応じて2段階で進める。

**第1段階：簡易統合（SendTo相当）**
- `argate.exe` にファイルパスを渡す形で、まず「送る」フォルダ相当の統合を実現
- 複数ファイル選択時の受け渡し処理をアプリ側で適切に処理する

**第2段階：コンテキストメニュー拡張**
- エクスプローラーの右クリックメニューに登録コマンドを動的サブメニューとして表示
- コマンドごとに「コンテキストメニューに表示する/しない」を設定可能
- 用途例：一括Zip解凍、ディレクトリ構造の一括作成、選択PDF結合など自作スクリプトの呼び出し

---

### マイルストーン4：ファイルマネージャー・AI連携深化

マイルストーン2bおよびマイルストーン3が安定してから着手。

**ファイルマネージャー（最小構成）**
- タブ付きブラウジング（セッション記憶）
- ブックマーク・最近開いたフォルダ
- コピー・移動・削除・リネーム・バッチリネーム
- インメモリインデックスによる高速ファジー検索
- プレビュー（テキスト・画像・コード・動画・音声・README）
- 仮想スクロール・フォルダサイズ非同期集計
- ウィジェットとしてワークスペースに組み込み可能

**AI連携深化**
- ファイルシステムインデックスをMCP経由でAIに公開
  - 「このフォルダの内容を読んでタグを自動付与」等の操作が可能
  - 「Projects/配下にXXプロジェクト作って」と指示するとフォルダ作成→監視フォルダが検知して自動登録
- セマンティック検索の追加（起動履歴・タグ・プロジェクト情報のベクトル化）
  - 例：「先週よく使ってたやつ」「このプロジェクトに関係するツール」
- MCPのファイル操作対応（コピー・移動・タグ付け等）

---

## 4. 横断機能：タグシステム

マイルストーン1から基盤を作り、段階的に対象を拡張する。

**対象**
- マイルストーン1〜：ランチャーの登録アイテム（アプリ・スクリプト・URL等）
- マイルストーン4〜：ファイル・フォルダ

**機能**
- 任意のタグを複数付与
- タグによるフィルタリング・検索
- タグに「デフォルト非表示」フラグを設定可能（センシティブなコンテンツの隠蔽）
- 非表示コンテンツはホットキーまたはパスワードでトグル表示

**設計方針**
- マイルストーン1の初期設計からSQLiteスキーマに組み込む
- マイルストーン4のファイル対応を見越したスキーマ設計

## 5. アイテムの実行コンテキスト定義

登録アイテムごとに設定できる実行パラメータ。マイルストーン1の設計時にSQLiteスキーマへ組み込む。マイルストーン3（コンテキストメニュー統合）のファイル受け渡しにも使用する。

**起動コマンド**

| 項目 | 内容 | 用途例 |
|---|---|---|
| `command` | 実行ファイルパス / URL | `blender.exe`、`powershell.exe` |
| `args` | 引数の配列（commandと分離） | `["-NoProfile", "-File", "start.ps1"]` |

commandとargsを分離することで、クォート処理・エスケープの責務をアプリ側で一元管理できる。

**引数テンプレート（ファイル受け渡し）**

コンテキストメニューやドラッグ＆ドロップでファイルを渡す際の形式を設定可能。

| モード | 動作 | 用途例 |
|---|---|---|
| `join=" "`（デフォルト） | スペース区切りで1引数にまとめる | 少数ファイルの一般的な受け渡し |
| `mode="file"` | 一時ファイル（paths.txt）にパスを書き出して渡す | 大量選択・パス長制限回避 |
| `mode="repeat"` | `--input <path>` をファイル数分繰り返す | 繰り返し引数が必要なCLIツール |

**その他パラメータ**

| 項目 | 内容 | 用途例 |
|---|---|---|
| Working Directory | 起動時のカレントディレクトリ。未指定の場合はコマンドのある場所を使用 | プロジェクトフォルダで Claude Code を起動 |
| 環境変数 | アイテムごとに上書き可能なキー・バリュー | Blender バージョンごとの設定パス切り替え |
| 管理者権限（UAC）フラグ | 管理者として実行するかどうか | レジストリ操作が必要なスクリプト |
| 表示名 / 検索用別名 | 検索時にヒットさせたい別名を複数設定可能 | `blen4` → Blender 4.x |
| 優先度 | ピン / 最近使った / 起動頻度（ピン＞頻度＞最近の順で表示） | コマンドパレットでの表示順制御 |

## 6. 共通基盤

マイルストーンをまたいで共有するモジュール。マイルストーン1の設計時から考慮する。

- **ファイル監視モジュール**：パス追跡（マイルストーン2）・ファイル検索インデックス（マイルストーン4）で共用
- **アイコン取得モジュール**：exe・URL両方で共通処理
- **SQLiteスキーマ**：マイルストーン4のファイル管理・AI連携まで見越した初期設計

---

## 7. 設計思想：プラグインアーキテクチャ

Argateのコアは「プラグインを読み込んで動かす仕組み」だけを持つ。全ての機能はプラグインとして実装する。

**原則**

- コアが直接持つのは「アイテムの登録・起動・常駐」のみ
- コマンドパレット・ワークスペース・MCP・ファイルマネージャーは全てプラグインとして実装する
- 公式プラグインとサードパーティプラグインは同一のインターフェースで動作する
- MCP・UI・CLIは同一のサービス層を叩く。MCPがUIや内部実装に直接触れる設計にしない

**メリット**

- 各機能が独立しているため、1つのプラグインのバグがコア全体に波及しない
- プラグイン単位でテスト・追加・削除ができる
- 将来のサードパーティ製プラグイン対応がコアの変更なしに実現できる

**マイルストーン1での扱い**

マイルストーン1はプラグイン構造なしのシンプルな実装で始める。ただしプラグインインターフェース（境界の定義）だけはマイルストーン1の設計時に確定させる。マイルストーン2移行時の全面リファクタを防ぐため。

---

## 9. 技術方針

### スタック選定理由

| レイヤー | 選択 | 理由 |
|---|---|---|
| UI | Svelte | 軽量・高速 |
| バックエンド | Rust | ファイル操作・検索・ログ処理の速度確保 |
| フレームワーク | Tauri | Electronより軽量、Rustと自然に統合 |
| データ | SQLite | ローカル完結・ゼロコスト |

---

## 10. 非機能要件

マイルストーンごとに優先する目標を分けて定義する。

**マイルストーン1（常駐・起動優先）**

| 項目 | 目標 | 測定条件 |
|---|---|---|
| 起動時間 | 2秒以内 | ホットキー押下 → UI表示までのP95レイテンシ |
| 常駐メモリ | 100MB以下 | 起動後30秒のIdle時 Working Set |
| バイナリサイズ | 20MB以下 | インストーラーなしの単体exe |

**マイルストーン4（検索・ファイル処理優先）**

| 項目 | 目標 | 測定条件 |
|---|---|---|
| ファイル検索速度 | 10万件規模で1秒以内 | インデックス構築後のクエリ応答時間 |
| 大量ファイル表示 | 仮想スクロールでUIが詰まらない | 1万件以上のフォルダでスクロール60fps維持 |

## 11. ユースケース・シナリオ

### マイルストーン1で解決されるシナリオ

**ゲーム起動の一元化**
Steam・DMM・ブラウザゲームなど起動元がバラバラなゲームを、このアプリ一箇所から起動できる。各ランチャーを開き直す手間がなくなる。

**Blender複数バージョン管理**
Blender 3.x・4.xなど複数バージョンをそれぞれ登録し、バージョンを選んで起動できる。インストール済みのバージョンを一箇所から選んで起動できればよく、ダウンロード・インストール管理は対象外。

**Claude Code・スクリプトのワンクリック起動**
PowerShellスクリプトやバッチファイルでClaude Codeを起動するコマンドを登録し、ボタン一つで実行できる。

**ActivityWatch起動**
ActivityWatch本体を登録しておき、他のツールと同じ場所から起動できる。

---

### マイルストーン2以降で実現するシナリオ

**ツール専用ワークスペースの活用**
例えば「Obsidianワークスペース」ページを作り、特定ボルトを開くコマンド・よく使うノートへのショートカット・関連スクリプトなどをそのページにまとめて登録しておく。ツールごとに操作をワークスペース単位で整理できる。

**Claude CodeからMCP経由で操作**
Claude Codeから「Blender起動して」「開発用ワークスペース作って」と自然言語で指示すると、MCPを通じてこのアプリが操作を実行する。

**コンテキストメニューから自作スクリプトを呼び出す（マイルストーン3）**
エクスプローラーでファイルを右クリックすると、登録済みの自作スクリプト（一括Zip解凍・PDF結合等）がメニューに表示される。複数ファイルを選択してそのまま渡せる。

**プロジェクトフォルダの作成（マイルストーン4）**
Claude Codeから「Projects/配下にXXプロジェクト作って」と指示すると、MCPを通じてアプリが実際にフォルダを作成し、監視フォルダ機能が検知して自動でランチャーに登録される。

---

## 12. マイルストーン計画

```
マイルストーン1（ランチャー単体）              → 登録・起動・常駐。毎日使えるものを最速で
マイルストーン2a（コマンドパレット拡張）       → 内蔵コマンド/CLI/カテゴリ名前空間
マイルストーン2b（ワークスペース）             → テンプレ＋固定ウィジェットから開始
マイルストーン2c（MCP）                       → まず起動・検索・登録のみ対応
マイルストーン3（コンテキストメニュー統合）     → SendTo相当→動的サブメニューの2段階
マイルストーン4（ファイル管理・AI深化）        → ファイルマネージャー・セマンティック検索
```

アクティビティログはActivityWatchに完全委譲。MCPサーバー経由でClaude Code連携も可能。

マイルストーン1完成時点で「毎日使えるもの」を目指す。

---

## 13. 未決事項・スコープ外

**未決事項**
- アプリ名・プロダクト名
- ゲームライブラリ（Steam API等）との自動同期をするかどうか

**基本設計への申し送り**
- **プラグインインターフェースの定義**：マイルストーン1はプラグイン構造なしで作るが、インターフェース定義だけはマイルストーン1設計時に行う。マイルストーン2移行時の全面リファクタを防ぐ
- **サービスレイヤーの設計**：MCP・UI・CLIが同一のサービス層を叩く構造を最初に確立する。MCPがUIや内部実装に直接触れる設計にしない
- **ファイル監視モジュールの抽象化**：パス追跡（マイルストーン2）とファイルインデックス（マイルストーン4）で二重実装しないよう、共通モジュールの境界をマイルストーン1設計時に決める
- **SQLite全マイルストーンスキーマ設計**：タグ・ファイルパス・ワークスペース・プラグイン情報のテーブル設計をマイルストーン1時点で全マイルストーン分見越して設計する
- **MCP権限境界**：自然言語による任意コマンド実行の窓になり得るため、許可制・スコープ制・確認UI・監査ログの思想をマイルストーン2設計時に組み込む
- **M3ファイル受け渡しの技術リスク**：複数ファイル選択時はOS→プロセス引数経由になるため、コマンドライン長制限やクォート処理の問題が出やすい。大量選択時は一時ファイル（paths.txt）経由で渡す逃げ道をM3設計時に組み込む

**コアに含めないがプラグインとして将来対応可能**
- Google Calendar / タスクリスト連携
- ActivityWatchワークスペース（使用時間の可視化）
- ターミナルエミュレータウィジェット（コマンド実行後の対話モード対応、Windows Terminal連携）
- その他サードパーティサービス連携

**スコープ外（現時点）**
- クラウド同期
- 他ユーザー向け配布・ライセンス販売
- Linux / macOS対応（構造だけ意識しておく）
