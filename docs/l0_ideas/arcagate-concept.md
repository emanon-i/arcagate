# Arcagate（アルカゲート）— Concept Document

---

## 1. 概要

PC上に散在する起動元（Steam、各種ランチャー、ブラウザ、スクリプト）を一箇所に集め、登録・整理・起動できる個人用コントロールパネル。コマンドパレット中心の軽量常駐アプリとして始まり、ワークスペース・ファイル管理・MCP連携へと段階的に育てる。

---

## 2. 課題と目的

### 課題

- アプリの起動元がバラバラ（Steam、DMMゲーム、ブラウザゲーム、各種ランチャー等）で一元管理できていない
- 同一アプリの複数バージョン（Blender 3.x / 4.x 等）の切り替えが煩雑
- PowerShellスクリプトやClaude Code起動などの定型操作に手数がかかる
- 既存のランチャー代替ツールは、モダンなUIでないか動作が重いか、開発ツールとゲームを同列に扱えない

### 目的

「起動元・種別を問わず、何でも登録して即座に起動できる」個人用ハブを構築する。毎日開くアプリとして常駐させ、起動の手間を削減する。起動ログは回数・最終日時の最小限のみ自前で持つ。

---

## 3. ユースケース

### M1で解決されるシナリオ

- **ゲーム起動の一元化**：普段はSteam/DMMと複数のランチャーを行き来しているが、Arcagateだけを開けば全ゲームにアクセスできる
- **Blender複数バージョン切り替え**：`blen3` / `blen4` と打ち分けるだけで目的のバージョンが起動する
- **Claude Codeのワンクリック起動**：PowerShellスクリプトをArcagateに登録し、コマンドパレットから即実行できる
- **ホットキー起動フロー**：どの画面にいてもホットキー一発でパレットが開き、名前を打ってEnterで起動する（これがM1のキラー体験）

### M2以降で実現するシナリオ

- **ツール専用ワークスペース**：「Obsidianワークスペース」ページにボルト別ショートカット・関連スクリプトを集約して管理する
- **Claude CodeからMCP経由で操作**：「Blender起動して」「開発用ワークスペース開いて」を自然言語で指示し、Arcagateが実行する
- **コンテキストメニューから自作スクリプト呼び出し（M3）**：エクスプローラーで複数ファイルを選択し右クリック→登録済みスクリプトを直接実行する
- **プロジェクト自動登録（M4）**：フォルダを作成すると監視フォルダ機能が検知し、ランチャーに自動登録される

## 4. 前提条件・制約

| 項目 | 内容 |
|---|---|
| コスト | ゼロコスト運用（OSS・無料サービスのみ） |
| プラットフォーム | Windows x86-64 優先、将来的なクロスプラットフォームを視野に入れた設計 |
| 技術スタック | Tauri + Svelte（UI）+ Rust（バックエンド）+ SQLite（ローカルDB） |
| 常駐メモリ | Idle時 Working Set 100MB以下（M1目標） |
| 起動時間 | ホットキー押下からUI表示まで P95 2秒以内（M1目標） |
| バイナリサイズ | インストーラーなし単体exe 20MB以下 |
| スコープ管理 | M1を最優先とし段階的に開発。M1完成前にM2に着手しない |

## 5. コアコンセプト

### 「登録できれば起動できる」—— 種別を問わない統一アイテムモデル

既存のランチャーは「ゲームランチャー」「開発ツール管理」「スクリプトランナー」が別アプリとして存在し、横断管理ができない。Arcagateは `.exe / URL / フォルダ / スクリプト / コマンド` を同一のアイテムモデルで扱い、「登録できれば何でも起動できる」を原則とする。これにより、Steam・ブラウザゲーム・PowerShellスクリプト・Claude Codeを単一の場所から扱える。

### コマンドパレット中心設計

UIの核はコマンドパレット（キーボードインターフェース）。マウスでも使えるが、「ホットキー → 名前を打つ → Enter」のフローが最速経路になるよう設計する。ワークスペース（M2b以降）はコマンドパレットを補完する位置付けであり、逆転させない。

### プラグインアーキテクチャ——コアは最小、機能は追加可能

コアが直接持つのは「アイテムの登録・起動・常駐」のみ。コマンドパレット拡張・ワークスペース・MCP・ファイルマネージャーはすべてプラグインとして実装する。公式プラグインとサードパーティプラグインは同一のインターフェースで動作する。M1はプラグイン構造なしで実装するが、インターフェース境界の定義だけはM1設計時に行い、M2移行時の全面リファクタを防ぐ。

### サービスレイヤーの一元化

MCP・UI・CLIは同一のサービス層を経由する。MCPがUIや内部実装に直接触れる設計にしない。これにより、自然言語指示でもUI操作でもCLIでも同じ操作結果になることを保証する。

## 6. 機能要件

マイルストーンごとに定義する。詳細はセクション8（マイルストーン）と合わせて参照。

### M1：登録・起動・常駐

- 任意の実行ファイル・URL・フォルダ・スクリプトを登録できる
- ラベル・アイコン・カテゴリ・検索用別名を設定できる
- exeのドラッグ&ドロップで即登録できる（アイコン自動取得付き）
- コマンドパレットから名前で検索・起動できる
- システムトレイに常駐し、ホットキーでどこからでも呼び出せる
- Windows起動時に自動起動できる
- 起動回数・最終起動日時をSQLiteに記録し、頻度・最近使ったもの順に表示できる
- SQLiteデータ一式をエクスポート/インポートできる
- 初回起動時のシンプルなセットアップウィザードがある
- タグに「デフォルト非表示」フラグを設定でき、ホットキーまたはパスワードでトグル表示できる（センシティブなコンテンツの隠蔽用途）

### M2a：コマンドパレット拡張

- CLIからコマンドパレットを操作できる（例：`arcagate run blender`）
- クリップボード履歴・スニペット・電卓を内蔵コマンドとして使える
- カテゴリにプレフィックスを設定し名前空間として使える（例：`gm` → Gamesカテゴリの絞り込み）
- ファイルシステム監視によるパス変更の自動追跡ができる

### M2b：ワークスペース

- ユーザーが任意のページを作成し、ウィジェットを自由に配置できる
- よく使うもの・最近使ったもの・プロジェクト一覧・監視フォルダを内蔵ウィジェットとして使える
- テーマ（ダーク・ライト・カスタムカラー）を切り替えられる
- UIの角丸・ボタンの丸み等のシェイプをテーマごとに設定できる
- テーマをインポート・エクスポートして共有できる
- 監視フォルダに追加されたアイテムをランチャーに自動登録できる
- Gitステータス（ブランチ名・未コミット変更・最終コミット日時）をプロジェクト一覧で確認できる

### M2c：MCP

- Arcagate自身がMCPサーバーを内蔵し、Claude Codeからアイテムの登録・起動・検索を自然言語で操作できる
- MCPセットアップ含め、Claude Codeへの指示だけで初期設定を完結できる

### M3：コンテキストメニュー統合

- エクスプローラーの「送る」相当の統合（第1段階）
- エクスプローラー右クリックメニューに登録コマンドを動的サブメニューとして表示（第2段階）
- 複数ファイル選択時の受け渡しに対応できる

### M4：ファイルマネージャー・AI連携深化

- タブ付きファイルブラウザ（セッション記憶・ブックマーク）として使える
- インメモリインデックスによる高速ファジー検索ができる（10万件規模で1秒以内）
- ファイルシステムインデックスをMCP経由でAIに公開し、タグ付与・フォルダ作成などを自然言語で操作できる
- セマンティック検索（「先週よく使ってたやつ」等）ができる

---

## 7. スコープ外

- クラウド同期（ローカル完結が原則）
- 他ユーザー向け配布・ライセンス販売（個人用途のみ）
- Linux / macOS ネイティブ対応（設計上は意識するが対応しない）
- Steamライブラリ等との自動同期（手動登録が基本だが、将来対応の可能性あり）
- ターミナルエミュレータ統合（PTY・エンコーディング作り込みは対象外。ログを別ウィンドウで開く程度に留める）
- Google Calendar・タスクリスト連携（将来プラグインとして対応可能だが初期は対象外）

## 8. マイルストーン / 実装ステップ

### M1：ランチャー単体

**目的：** 「毎日使えるもの」を最速で作る。起動体験のコアを確立する。

**成功条件：**

- ゲーム起動・Blender複数バージョン切り替え・Claude Code起動の3ユースケースをノーエラーで実行できる
- ホットキー → 名前入力 → Enter → 起動のフローを10回繰り返してエラーが発生しない
- 常駐時メモリ 100MB以下、ホットキー起動から表示まで P95 2秒以内を計測で確認できる

**注記：** M1はプラグイン構造なしで実装する。ただしプラグインインターフェースの境界定義（どこで切り分けるか）はM1設計時に確定させる。

---

### M2a：コマンドパレット拡張

**依存：** M1完成・安定後
**目的：** パワーユーザー向けの操作を強化し、単なるランチャーから脱却する。

**成功条件：**

- CLIからアイテム起動できる
- クリップボード履歴・スニペット・電卓が内蔵コマンドとして呼び出せる
- カテゴリプレフィックスで絞り込み起動できる

---

### M2b：ワークスペース

**依存：** M2a完成後
**目的：** 「ホーム画面」を作り、ツールとプロジェクトをビジュアルに整理できるようにする。

**成功条件：**

- 2つ以上のページを作成し、内蔵ウィジェットを配置して日常運用できる
- 監視フォルダが新規ファイルを検知してアイテムを自動登録できる

---

### M2c：MCP

**依存：** M2b完成後（並行着手可）
**目的：** Claude CodeからArcagateを自然言語で操作できるようにする。

**成功条件：**

- Claude Codeから「〇〇を起動して」「〇〇を登録して」を実行できる
- MCPサーバーのセットアップをClaude Codeへの指示だけで完結できる

---

### M3：コンテキストメニュー統合

**依存：** M2a完成・安定後
**目的：** ファイル操作をArcagateの登録コマンドと直接繋ぐ。

**成功条件：**

- SendTo相当の統合でファイルをスクリプトに渡せる（第1段階）
- 右クリックメニューに登録コマンドが表示され、複数ファイル選択して渡せる（第2段階）

---

### M4：ファイルマネージャー・AI連携深化

**依存：** M2bおよびM3完成・安定後
**目的：** ファイル操作を統合し、AIとの連携を深める最終形態。

**成功条件：**

- タブ付きファイルブラウザが日常のエクスプローラー代替として使える
- 10万件規模のファイル検索が1秒以内で応答する
- Claude CodeからMCP経由でファイル操作（タグ付け・フォルダ作成）ができる

---

## 9. 継続判断基準

### 継続する条件

- M1完成後、2週間以上毎日自分が使い続けている
- ホットキー起動からUI表示のP95が常時2秒以内に収まっている
- 常駐メモリが100MB以下を維持している

### 再検討する条件

- M1完成後2週間で自発的に使わなくなった場合 → コアUXの再定義を検討
- Tauriのアップデートや依存ライブラリの破壊的変更で維持コストが急増した場合 → スタック見直しを検討
- M2のプラグイン移行でコア全体の大規模リファクタが必要になった場合 → M1のインターフェース設計に立ち返る

## 10. 技術スタック概要

| レイヤー | 選択 | 理由 |
|---|---|---|
| UIフレームワーク | Svelte | 軽量・高速・バンドルサイズ小 |
| バックエンド | Rust | ファイル操作・検索・ログ処理の速度確保 |
| デスクトップフレームワーク | Tauri | Electronより軽量、Rustと自然に統合 |
| ローカルDB | SQLite | ローカル完結・ゼロコスト・全マイルストーン共通 |

**SQLiteスキーマについて：** タグ・ファイルパス・ワークスペース・プラグイン情報のテーブルはM1時点で全マイルストーン分を見越して設計する。後から追加するとマイグレーションコストが高い。

## 12. 懸念事項

- **M3：コマンドライン長制限**（M3設計時に対処）
  複数ファイル選択時はOS→プロセス引数経由になるため、パス長制限やクォート処理問題が発生しやすい。大量選択時は一時ファイル（paths.txt）経由の逃げ道をM3設計時に組み込む。

- **M2：MCP権限境界**（M2c設計時に対処）
  MCPが自然言語による任意コマンド実行の窓になり得る。許可制・スコープ制・確認UI・監査ログの思想をM2c設計時に明示的に組み込む。

- **M2：プラグイン移行リファクタのリスク**（M1設計時に対処）
  M1をプラグイン構造なしで作った場合、M2移行時に大規模リファクタが発生するリスクがある。インターフェース境界（コアとプラグインの切り分け）だけはM1設計時に確定させることで軽減する。

- **M4：ファイルインデックスとパス監視の二重実装**（M1設計時に対処）
  パス追跡（M2）とファイルインデックス（M4）で同種の処理が必要になる。共通のファイル監視モジュールを最初から設計しておき、二重実装を防ぐ。

- **Tauriバージョン依存**
  Tauriのアップデートに追従できないと全体が止まる。Tauri本体のAPIに直接依存する部分を最小化し、変更に強い構造を維持する。

- **Steam API等ゲームライブラリ自動同期（未決）**
  手動登録が基本方針だが、Steamライブラリの自動同期（インストール済みゲームの自動取り込み）をするかどうかは判断保留。対応する場合はM2以降でプラグインとして実装する方針。

## 13. 設計思想

### 全機能はプラグイン——コアは「動かす仕組み」だけを持つ

コアが直接持つのは「アイテムの登録・起動・常駐」のみ。コマンドパレット・ワークスペース・MCP・ファイルマネージャーはすべてプラグインとして実装する。公式プラグインとサードパーティプラグインは同一のインターフェースで動作する。この原則により、1つのプラグインのバグがコアに波及しない。

### サービス層の一元化——MCP・UI・CLIは同じ入口を使う

MCP・UI・CLIはサービス層という共通インターフェースを経由する。MCPがUIや内部実装に直接触れる設計にしない。これにより、自然言語からの操作とUI操作が同一結果を保証でき、テストも可能になる。

### 初期設計に組み込む共通基盤

M1時点で以下の3点をスキーマ・インターフェース設計に折り込む。後から追加するコストが特に高い領域のため：

- **SQLiteスキーマの全マイルストーン設計**：タグ・ファイルパス・ワークスペース・プラグイン情報のテーブル
- **ファイル監視モジュールの抽象化**：パス追跡（M2）とファイルインデックス（M4）で共用できる共通境界の定義
- **プラグインインターフェースの境界定義**：M1は使わないが、M2移行時の全面リファクタを防ぐために定義だけ確定させる
