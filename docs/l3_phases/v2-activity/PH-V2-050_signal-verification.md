# PH-V2-050: Phase 0 — 信号取得の事実検証 & 方式最適化 (ゲート)

## 目的

以降の全実装フェーズ (100–800) が前提にする **「各信号の実証済み取得方式」を確定する**ゲート。L2 で
予定した取得方式 (USN / RDCW / SMTC / GetLastInputInfo / UIA 等) を**鵜呑みにせず**、(1) 実機で本当に
取れるかを生出力で裏取りし、(2) より良い取得方式が無いかを調査・実測比較し、負荷最優先で最良を選ぶ。
これは**本実装ではない** — 読み取り中心・使い捨てスパイクによる検証と、確定表の産出に限る。

前セッションの実機スパイク (activity-recorder.md 実機表 2026-07-02) を正式フェーズ化・拡張する。取れない
信号は L2 から落とすか代替に差し替える判断もここで行い、必要なら L2 spec 更新を申し送る。

## スコープ

### やること (1) 各信号の実機検証 — 生出力で確定

L2 予定の各信号について、実機で「取れる / 取れない / admin 要否 / 粒度 / 取りこぼし特性」を**生出力
(raw サンプル)** で確定する。前回未検証・要再検証を優先:

| 信号                  | L2 予定方式                        | 検証で確定すること                                                                                             |
| --------------------- | ---------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| 前面窓 + title + path | `GetForegroundWindow` + proc path  | hwnd/pid/proc/実行イメージパス/title の取得可否・粒度                                                          |
| 実操作 idle           | `GetLastInputInfo`                 | idle_ms の取得可否・分解能                                                                                     |
| 再生メディア          | SMTC `SessionManager.RequestAsync` | 全プレイヤー横断の app/title/artist 取得、停止時挙動                                                           |
| ファイル操作 (本命)   | USN `FSCTL_READ_USN_JOURNAL`       | **admin 要否の確定**・reason flag 粒度・fileID→path 解決・wrap/欠損特性                                        |
| ファイル操作 (代)     | ReadDirectoryChangesW              | 非 admin 捕捉範囲・バッファ溢れ特性・重複                                                                      |
| **ブラウザ URL**      | UIA アドレスバー                   | **前回ロック中で未検証 → unlocked browser で必ず再検証**。主要 3 + Chromium 系で取得可否・ドメイン粒度・失敗率 |

- 各検証は生出力を残す (「取れた」の主張でなく実データ)。admin 要否は昇格有無の両方で実測して確定
- ブラウザ URL は unlocked で明確に決着させ、**core 格上げ or 候補据え置きを根拠付きで**判定する

### やること (2) 「もっといい取り方」の調査・実測比較 (重要)

現状 L2 の方式を最良と仮定せず、各信号で**より良い取得方式が無いか**を調査し実測比較する。比較観点は
**負荷 (PC に絶対負荷をかけない) を最優先軸**に、粒度 / 確実性 (取りこぼし) / 権限の軽さ / 保守性:

| 信号                            | 比較する候補方式 (これに限らない)                                                                                                    |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| ファイル操作                    | USN / RDCW / **ETW (Kernel FileIO provider)** / minifilter。ETW は負荷・権限で USN に勝てるか実測 (常時 file trace は負荷制約に注意) |
| 前面窓                          | `SetWinEventHook(FOREGROUND)` (event) vs ポーリング。event 駆動の取りこぼし・負荷                                                    |
| 実操作 / idle                   | `GetLastInputInfo` vs raw input / WM_INPUT vs ETW input。分解能 vs 負荷                                                              |
| メディア                        | SMTC vs per-app 経路 (browser / player 個別)。横断性 vs 依存                                                                         |
| ブラウザ URL                    | UIA vs 各ブラウザの automation / accessibility (CDP / native a11y など) vs 拡張。拡張ゼロ制約と粒度の両立                            |
| システムメトリクス全体 (800 用) | 全体 CPU / RAM / disk / network の取得。性能カウンタ / WMI / ETW / `GetSystemTimes` 系。サンプリング間隔と負荷を実測                 |
| プロセス別リソース (800 用)     | アプリ別 CPU / RAM。性能カウンタ / ETW / WMI / `GetProcessTimes`・`NtQuerySystemInformation`。負荷・権限で最良を選ぶ                 |

- L2 の方式より良いものが見つかれば **L3 で採用方針を示し**、必要なら L2 spec 更新を申し送る
- 負荷が読めない候補 (ETW / WMI 等) は短時間スパイクで実測し、数値で採否を決める (印象で採らない)

### やること (3) 確定表の産出

Phase 0 の成果物 = **「各信号の実証済み取得方式」確定表**: 信号 / 採用方式 / 根拠 (実測負荷・粒度・権限・
取りこぼし) / 代替との比較結論 / L2 との差分 (据え置き / 変更 / 削除 / 差し替え)。これを 100–800 の前提にする。
確定表は `docs/l3_phases/v2-activity/audit/` に置き、変更が要る L2 spec は申し送りとして列挙する。

### やらないこと

- **本実装をしない** — recorder / collector / store の production コードを書かない。スパイクは使い捨て
  (throwaway)。検証で作ったコードを実装に流用しない (実装は 100–800)
- **負荷をかける方式で長時間走らせない** — 全走査 / per-file hash / 常時カーネルトレースの本番運用検証は
  しない (短時間スパイクで負荷傾向を測るに留める)。負荷制約 (REQ-20260702-003) をスパイクでも守る
- **キー内容 / UIA 汎用スクレイプを検証対象にしない** — idle は量のみ、UIA はブラウザ URL + 既知アプリ
  ドキュメント名の小集合のみ (禁止事項は検証段階から守る)
- **確定していない方式を L2 の正本として書き換えない** — 変更は申し送りに留め、L2 更新は別途 user 判断

## 依存

- 先行: なし (V2 の最初・ゲート)
- 後続: **100–800 の全実装フェーズ**。各フェーズは Phase 0 の確定表に載った採用方式を前提に実装する

## 受け入れ条件 (各信号 実証 or 代替確定)

- [ ] 前面窓 / GetLastInputInfo / SMTC / USN / RDCW の各信号が **生出力サンプル付き**で
      「取れる / 取れない / admin 要否 / 粒度 / 取りこぼし特性」が確定している
- [ ] **ブラウザ URL が unlocked browser で再検証**される。検証マトリクスを明示 (Chrome / Edge / Firefox +
      導入済 Chromium 系 1 種、各ブラウザで前面化 / タブ切替の複数回試行、通常 / プライベート窓の別)。
      取得可否・ドメイン粒度・**取得成功率を試行ログ (CSV/JSON 生出力) で確定**し、core 格上げの成功率閾値を
      050 が明記した上で 格上げ / 候補据え置きを判定する
- [ ] システムメトリクス**全体 (CPU/RAM/disk/net) とプロセス別**の取得方式が、source API / サンプリング間隔 /
      権限 / 生出力サンプル / 実測負荷付きで確定している (800 の前提)
- [ ] USN の **admin 要否が昇格有無の両方の実測で確定**し、wrap / 欠損 / fileID→path 解決の特性が記録されている
- [ ] 各信号で L2 方式と代替候補 (最低: ファイル操作・ブラウザ URL・プロセス別リソースの 3 領域) が
      **負荷最優先で実測比較**され、採用方式と却下理由が数値・生出力で示されている
- [ ] 取れない / 粒度不足の信号について「L2 据え置き / 変更 / 削除 / 代替差し替え」の判断が下されている
- [ ] 成果物「実証済み取得方式 確定表」が `audit/` に産出され、L2 spec 更新の申し送りが列挙されている
- [ ] スパイクが使い捨てで production コードを含まない — **機械検出**: 050 の成果物 commit で
      `git diff --name-only -- src src-tauri` が空 (スパイクは `docs/l3_phases/v2-activity/audit/` または
      `scratch/` 配下のみ、production path を触らない)

## 検証方針

- 実機は agent dev で user 実機に準じた環境。admin 検証は昇格 helper を立てて両条件で実測 (`agent-self-complete`)
- 負荷は実 target・実 disk・能動シナリオで計測 (warm fixture の再現漏れを避ける、`perf-audit-before-measure`)
- 「取れた」の判定は生出力を Read で読み返して確認 (主張でなくデータで、`dom-not-fixed` 同精神)
- 代替方式は同一シナリオで横並び計測し、負荷 → 粒度 → 確実性 → 権限 → 保守性の順で採否を決める

## リスク

- **スパイクが本実装化する**: 検証コードを流用すると設計固定枠を外れる → throwaway を要件に固定、実装は 100–800
- **負荷計測の環境ズレ**: warm fixture で軽く見え実機で重い → 実 disk・能動シナリオで計測
- **ブラウザ UIA の脆さ**: バージョンで tree が変わる → unlocked で per-browser の取得可否を実測し、失敗率も確定
- **ETW/WMI が重い**: 低負荷と誤認して採用すると負荷制約を割る → 数値実測で採否、印象で採らない
- **確定を先送り**: 「たぶん取れる」で 100 に進むと下流が崩れる → 生出力での確定を gate 条件にする

## 横展開

- 確定表は 100 (どの列を持つか) / 200 (窓/実操作/メディアの方式) / 300 (USN/RDCW の方式・admin) /
  800 (プロセス別リソースの方式) の前提として直接参照される
- ブラウザ URL の判定は 200 の「候補据え置き」記述と整合させる (格上げなら 200/400 に反映を申し送り)
- L2 spec 更新の申し送りは activity-recorder.md 実機表の追補として扱う (履歴 mutation でなく追記)
