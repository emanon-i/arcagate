# PH-419: 実測ベンチ取得（startup / idle-mem / search） — PH-402 deferred 解消

## 問題

engineering-principles.md §9 の数値ベンチマーク表で **未計測** が残存:

| 指標                          | Arcagate 閾値         | 業界標準               | 現状       |
| ----------------------------- | --------------------- | ---------------------- | ---------- |
| ホットキー → パレット表示 P95 | 2 秒以内              | Raycast/Alfred ≤ 100ms | **未計測** |
| 検索 → 結果反映 P95           | 150ms debounce + 即時 | Raycast < 50ms         | **未計測** |
| Idle メモリ                   | 100MB 以下            | Raycast ~100MB         | **未計測** |
| クラッシュ率                  | 0 件目標              | 0.1% 未満              | **未計測** |

batch-87 PH-382/383 で `scripts/bench/startup.ps1` / `idle-memory.ps1` を新設したが PH-402 で計測実行が deferred になった。本 plan で実行 + ベースライン化。

## 改修

### 1. startup ベンチ実行

`scripts/bench/startup.ps1` 実行:

- 10 回連続でアプリを起動 → ホットキー押下 → パレット表示
- Performance.now() ベースで時間計測（webview 内 console log → log file）
- P50 / P95 / max を `docs/l2_architecture/performance-baseline.md` に追記

### 2. idle メモリベンチ実行

`scripts/bench/idle-memory.ps1` 実行:

- アプリ起動 → 5 分間 idle → `Get-Process arcagate | Select WorkingSet64` 計測
- 5 回繰り返し → P50 / P95 を記録

### 3. 検索体感ベンチ（新設）

`scripts/bench/search-latency.ps1` 新設:

- アプリ起動 → palette 開く → 文字入力 → 結果表示までの時間
- 10 件 / 100 件 / 1000 件のアイテム数で計測
- 計測手段: webview console log + Rust 側 tracing

### 4. baseline 確定 + violations 検出

`docs/l2_architecture/performance-baseline.md` を **正式 baseline** として確定:

- 各指標の baseline 値 + 業界標準 + Arcagate 閾値
- 違反項目があれば **次バッチ Plan 候補** として dispatch-log に記録
- baseline 超過を CI で検出する仕組み（後続 batch で）

### 5. engineering-principles.md §9 表更新

「**未計測**」を実測値に置換:

```markdown
| ホットキー → パレット表示 P95 | 2 秒以内 / **実測 XXms** | Raycast/Alfred ≤ 100ms |
```

達成 / 未達を ✅ / ⚠️ / 🔴 で明示。
