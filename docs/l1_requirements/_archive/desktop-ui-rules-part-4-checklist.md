# Desktop UI Rules Part 4 — 完了チェックリスト

[desktop-ui-rules.md](./desktop-ui-rules.md) のチェックリスト集。

## 1. 目的確認

- この画面の主目的は一言で説明できるか
- その目的に対して UI が過不足なく支援しているか

## 2. 状態確認

- 処理中・成功・失敗・選択状態が分かるか

## 3. 立て直し確認

- ミスしたときに戻れるか、直せるか、やり直せるか

## 4. 認知負荷確認

- 最初の一手が分かるか
- 読み順と関係性が自然に見えるか

## 5. 一貫性確認

- 既存画面や既存部品と矛盾していないか
- OS の慣習から大きく外れていないか

## 6. アクセシビリティ確認

- キーボード操作、フォーカス可視性、ラベル、色依存の有無を確認したか

## 7. 効率確認

- 頻出操作を不必要に重くしていないか

## 8. 最後に視覚整理

- 余白、整列、コントラスト、ノイズを整理したか

# 5. 変更理由の書き方

実装や修正の説明では、次の形で理由を残す。

```md
### UI/UX Review Note

- Goal: この画面/変更で達成したいこと
- Principle: どの原則に基づく判断か
- Reason: なぜその変更が必要か
- Before: 変更前に何が分かりにくかったか / 使いづらかったか
- After: 変更後に何が分かりやすくなったか / 使いやすくなったか
- Trade-off: 何を残し、何を優先したか
```

### 例

```md
### UI/UX Review Note

- Goal: 送信画面で二重送信を減らす
- Principle: P1 状態は常に見えるようにする
- Reason: 押下後に無反応に見え、再クリックを誘発していたため
- Before: 送信中かどうか分からず不安になる
- After: 送信中表示とボタン状態変化で進行が分かる
- Trade-off: 軽微な処理でも状態表示が増えるが、誤認防止を優先した
```

# 6. 要約

この文書の目的は、エージェントに「勝手に止まれ」と命じることではない。
目的は、**実装した UI を自分で見直し、なぜその形がよいのかを説明しながら改善できるようにすること** である。

そのために、エージェントは次を意識する。

- まず操作できることを守る
- 次に状態と結果を見えるようにする
- 失敗しても立て直せるようにする
- 一貫性と OS 文脈を尊重する
- アクセシビリティを操作性の一部として扱う
- 最後に視覚的な品質を整える

そして各変更について、
**「どの原則に基づく判断か」「なぜそれが必要か」「何が良くなったか」** を説明できるようにする。

以上を、デスクトップ UI/UX の実装・自己レビュー原則とする。
