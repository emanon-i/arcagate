# 0. この文書の使い方

この文書は、次の 2 段階で使う。

## 実装前

- いま作る画面の主目的は何か
- ユーザーがその画面で最初にやることは何か
- 失敗すると何が困るか
- 既存の OS / アプリ文脈で期待される挙動は何か

を整理する。

## 実装後

- 実装した UI が目的達成を助けているか
- 情報不足や強調不足で迷わせていないか
- 見た目を整える代わりに操作性を落としていないか
- 変更前より理解しやすく、直しやすく、使いやすくなっているか

を見直す。

この文書は「設計ポリシーの押し付け」ではなく、
**変更の妥当性を説明できるかどうかを確認するための基準**として使う。

# 1. 最上位の判断原則

UI/UX の判断で迷った場合は、次の順で優先する。

1. **操作できること**
2. **結果が分かること**
3. **失敗しても立て直せること**
4. **意味が理解できること**
5. **一貫していて予測できること**
6. **素早く使えること**
7. **最後に見た目が整っていること**

### なぜこの順序か

デスクトップアプリでまず重要なのは、見た目の洗練よりも
**操作不能・誤認・誤操作・復旧不能を避けること** だからである。

視覚的な美しさは重要だが、
それは操作可能性や理解可能性を満たした後に初めて意味を持つ。

### この順序を無視したときに起きやすいこと

- 見た目を整えた結果、どこを押すべきか分からなくなる
- 画面がミニマルになった代わりに状態や失敗理由が見えなくなる
- 統一感のために OS の慣習を壊し、使いづらくなる

# 2. 原則一覧

各原則は次の観点で書く。

- **原則**: 何を意識すべきか
- **理由**: なぜそれが必要か
- **見落としサイン**: うまくいっていないときに起きやすい現象
- **自己レビュー**: 実装後に自分へ問う質問

---

# 3. 原則

## P1. 状態は常に見えるようにする

### 原則

ユーザーは、今その画面で何が起きているかを追跡できるべきである。
処理中・成功・失敗・選択中・現在位置・モード状態は、必要に応じて可視化する。

### 理由

状態が見えないと、ユーザーは「押せていない」「固まった」「失敗した」と誤認する。
その結果、二重操作、離脱、不信感が生まれる。

### 見落としサイン

- クリック後に反応がなく見える
- 成功したのか失敗したのか外見上分からない
- どのタブ・項目・モードが有効か分からない

### 自己レビュー

- 押した直後に「何か起きた」と分かるか
- 完了時に結果を読み取れるか
- エラー時に、成功時と明確に区別できるか
- 現在地や選択状態を迷わず把握できるか

---

## P2. 失敗は前提として扱い、立て直しやすくする

### 原則

良い UI は、失敗をゼロにすることではなく、
**失敗しても戻れる・直せる・やり直せる** ように作られている。

### 理由

入力ミス、誤クリック、通信失敗、操作の勘違いは必ず起きる。
問題は失敗の有無ではなく、失敗後にユーザーが止まるか、立て直せるかである。

### 見落としサイン

- 一度のミスで最初からやり直しになる
- 削除や送信の影響範囲が分からない
- エラー後に次に何をすればよいか分からない

### 自己レビュー

- 誤操作しても回復できるか
- 失敗したときの次の一手が見えるか
- ユーザーは「何を失ったか」「何が残っているか」を理解できるか

---

## P3. 主要操作はすぐ見つかるようにする

### 原則

各画面には、その画面の主目的に対応する主要操作があり、
それが他の要素に埋もれないようにする。

### 理由

ユーザーは画面を見るたびに全要素を精査したいわけではない。
多くの場合、「次に何をすればよいか」を素早く把握したい。
主操作が埋もれると、迷いと誤操作が増える。

### 見落としサイン

- 重要なボタンが他と同じ強さで並んでいる
- どれが主操作でどれが補助操作か分からない
- 危険な操作が主操作に見えてしまう

### 自己レビュー

- 初見のユーザーが最初の一手を推測できるか
- 主要操作と補助操作の見た目に差があるか
- 重要な操作が情報ノイズに埋もれていないか
