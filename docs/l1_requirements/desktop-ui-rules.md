# デスクトップソフトウェア UI/UX 実装・自己レビュー原則

> 目的: コーディングエージェントがデスクトップアプリを実装したあと、
> 自分の変更を **UI/UX の観点で見直し、必要なら自力で修正できる** ようにするための原則集。
>
> この文書は、禁止事項の一覧ではない。
> **判断のための原則と、その原則が必要な理由** を与えるための文書である。
> したがって、エージェントは本文を「止まるため」ではなく、
> **実装方針を選ぶこと・実装後に見直すこと** の両方に使う。

---

# 0. この文書の使い方

この文書は、次の 2 段階で使う。

## 実装前

- いま作る画面の主目的は何か
- ユーザーがその画面で最初にやることは何か
- 失敗すると何が困るか
- 既存の OS / アプリ文脈で期待される挙動は何か

を整理する。

## 実装後

- 実装した UI が目的達成を助けているか
- 情報不足や強調不足で迷わせていないか
- 見た目を整える代わりに操作性を落としていないか
- 変更前より理解しやすく、直しやすく、使いやすくなっているか

を見直す。

この文書は「設計ポリシーの押し付け」ではなく、
**変更の妥当性を説明できるかどうかを確認するための基準**として使う。

---

# 1. 最上位の判断原則

UI/UX の判断で迷った場合は、次の順で優先する。

1. **操作できること**
2. **結果が分かること**
3. **失敗しても立て直せること**
4. **意味が理解できること**
5. **一貫していて予測できること**
6. **素早く使えること**
7. **最後に見た目が整っていること**

### なぜこの順序か

デスクトップアプリでまず重要なのは、見た目の洗練よりも
**操作不能・誤認・誤操作・復旧不能を避けること** だからである。

視覚的な美しさは重要だが、
それは操作可能性や理解可能性を満たした後に初めて意味を持つ。

### この順序を無視したときに起きやすいこと

- 見た目を整えた結果、どこを押すべきか分からなくなる
- 画面がミニマルになった代わりに状態や失敗理由が見えなくなる
- 統一感のために OS の慣習を壊し、使いづらくなる

---

# 2. 原則一覧

各原則は次の観点で書く。

- **原則**: 何を意識すべきか
- **理由**: なぜそれが必要か
- **見落としサイン**: うまくいっていないときに起きやすい現象
- **自己レビュー**: 実装後に自分へ問う質問

---

# 3. 原則

## P1. 状態は常に見えるようにする

### 原則

ユーザーは、今その画面で何が起きているかを追跡できるべきである。
処理中・成功・失敗・選択中・現在位置・モード状態は、必要に応じて可視化する。

### 理由

状態が見えないと、ユーザーは「押せていない」「固まった」「失敗した」と誤認する。
その結果、二重操作、離脱、不信感が生まれる。

### 見落としサイン

- クリック後に反応がなく見える
- 成功したのか失敗したのか外見上分からない
- どのタブ・項目・モードが有効か分からない

### 自己レビュー

- 押した直後に「何か起きた」と分かるか
- 完了時に結果を読み取れるか
- エラー時に、成功時と明確に区別できるか
- 現在地や選択状態を迷わず把握できるか

---

## P2. 失敗は前提として扱い、立て直しやすくする

### 原則

良い UI は、失敗をゼロにすることではなく、
**失敗しても戻れる・直せる・やり直せる** ように作られている。

### 理由

入力ミス、誤クリック、通信失敗、操作の勘違いは必ず起きる。
問題は失敗の有無ではなく、失敗後にユーザーが止まるか、立て直せるかである。

### 見落としサイン

- 一度のミスで最初からやり直しになる
- 削除や送信の影響範囲が分からない
- エラー後に次に何をすればよいか分からない

### 自己レビュー

- 誤操作しても回復できるか
- 失敗したときの次の一手が見えるか
- ユーザーは「何を失ったか」「何が残っているか」を理解できるか

---

## P3. 主要操作はすぐ見つかるようにする

### 原則

各画面には、その画面の主目的に対応する主要操作があり、
それが他の要素に埋もれないようにする。

### 理由

ユーザーは画面を見るたびに全要素を精査したいわけではない。
多くの場合、「次に何をすればよいか」を素早く把握したい。
主操作が埋もれると、迷いと誤操作が増える。

### 見落としサイン

- 重要なボタンが他と同じ強さで並んでいる
- どれが主操作でどれが補助操作か分からない
- 危険な操作が主操作に見えてしまう

### 自己レビュー

- 初見のユーザーが最初の一手を推測できるか
- 主要操作と補助操作の見た目に差があるか
- 重要な操作が情報ノイズに埋もれていないか

---

## P4. 同じ意味のものは同じように扱う

### 原則

同じ意味・同じ役割のものは、
画面が変わってもできるだけ同じラベル・同じ挙動・同じ見え方で扱う。

### 理由

一貫性は、ユーザーが「ここでもたぶん同じだろう」と予測する助けになる。
予測可能性が上がると、再学習のコストが減る。

### 見落としサイン

- 似た機能に別の名前が付いている
- 画面ごとに保存・閉じる・戻るの挙動が違う
- 類似コンポーネントが微妙に異なるスタイルで乱立している

### 自己レビュー

- この操作は、別の画面でも同じ意味で使われているか
- 既存の共通コンポーネントやトークンに寄せられるか
- 今回だけの都合で新しい表現を増やしていないか

### 補足: ラベルはアイコン名ではなく機能・状態・アクションを書く

ボタンやチップの **ラベル文字** は、隣のアイコンの形状名・記号名ではなく、
押したときに何が起きるか / 現在どの状態かを書く。

| 違反             | 正                  | 理由               |
| ---------------- | ------------------- | ------------------ |
| ⭐ + 「星」      | ⭐ + 「お気に入り」 | 「星」はアイコン名 |
| ☰ + 「三本線」   | ☰ + 「メニュー」    | 同上               |
| ＋ + 「プラス」  | ＋ + 「追加」       | 同上               |
| 🔍 + 「虫眼鏡」  | 🔍 + 「検索」       | 同上               |
| 🗑 + 「ゴミ箱」   | 🗑 + 「削除」        | 同上               |
| ⚙ + 「歯車」     | ⚙ + 「設定」        | 同上               |
| ✓ + 「チェック」 | ✓ + 「完了」        | 同上               |

書き分け:

- アクションボタン → **動詞**（追加 / 削除 / 編集 / 起動 / 検索）
- 状態切替 → **状態名**（お気に入り / 表示 / 非表示）
- ナビゲーション → **行き先名**（設定 / ライブラリ / ホーム）

`aria-label` も同じ。スクリーンリーダーで「星ボタン」と読まれても何の機能かわからない。

### 補足 2: 同じ機能には同じアイコン + 同じラベル（画面間の一貫性）

「Library 機能」を表すアイコンとラベルは、本体タブでも Settings サイドバーでも同じものを使う。
画面ごとに別アイコン・別ラベルを書くと、ユーザーは別機能だと誤認する。

実装ルール:

- 共有ナビ項目は `src/lib/nav-items.ts` で 1 箇所定義
- 各画面 (TitleBar / SettingsPanel / Sidebar 等) は import して参照
- 直接 lucide icon を import + label 文字列直書きは禁止（ナビ項目に限る）

### 補足 3: 横展開原則（不整合を見つけたら全画面チェック）

1 つの不整合を発見したら、同種パターンが他の画面にも存在する前提で全画面 audit:

- 全 `<button>` `<a>` `<TitleTab>` `<TitleAction>` `<MenuItem>` の icon + label が機能を説明しているか
- 同じ機能を扱う画面間で icon / label が揃っているか
- 言語混在（英 + 日）が起きていないか

1 ファイル fix で終わらせない。同じ間違いは必ず他にもある。

---

## P5. OS とアプリの文脈に逆らわない

### 原則

デスクトップアプリは単独で存在するのではなく、
Windows / macOS の文脈の中で使われる。
そのため、ショートカット、ウィンドウ挙動、ファイル操作、テキスト編集などは、
OS の期待にできるだけ沿う。

### 理由

デスクトップでは、ユーザーは OS 全体で身についた筋記憶や期待を持っている。
そこから外れると、違和感ではなく実際の操作ミスにつながる。

### 見落としサイン

- Ctrl / Cmd の扱いが曖昧
- テキスト選択や右クリックの挙動が独特
- ウィンドウ移動・リサイズ・保存ダイアログが期待と違う

### 自己レビュー

- この挙動は OS の標準期待と大きくずれていないか
- キーボードショートカットは Windows / macOS で自然か
- テキスト編集やファイル操作で変な癖を作っていないか

---

## P6. アクセシビリティは追加要件ではなく操作性そのものと考える

### 原則

アクセシビリティは「余裕があれば足すもの」ではなく、
その UI が本当に操作可能かどうかを決める中核要素として扱う。

### 理由

フォーカスが見えない、ラベルが失われている、色だけで状態を示す、といった問題は、
一部ユーザーだけの問題ではなく、UI の理解可能性と操作可能性そのものを下げる。

### 見落としサイン

- キーボードだけで主要操作に辿れない
- フォーカス位置が見えない
- アイコンだけで意味を伝えようとしている
- ライト/ダークの一方で読みにくい

### 自己レビュー

- Tab 移動だけでも主要フローを進められるか
- フォーカスがどこにあるか一目で分かるか
- 色が見分けにくくても状態を理解できるか
- ラベルや名前が削られていないか

---

## P7. フォームは「何をどう入れればよいか」が持続的に分かるようにする

### 原則

フォームでは、項目名、制約、単位、必須性、エラー箇所が
入力中も失われないようにする。

### 理由

フォームではユーザーは記憶ではなく画面上の手掛かりに頼って入力する。
項目名や制約が消えると、入力負荷とエラー率が上がる。

### 見落としサイン

- 入力を始めた途端に項目名が見えなくなる
- 送信するまで制約が分からない
- エラーがどの項目に対応するか分からない

### 自己レビュー

- 入力後も何の欄か分かるか
- 必須/任意や書式制約が必要な場所で見えているか
- エラー時にどこを直せばよいかすぐ分かるか

---

## P8. 情報は読み順と関係性が見えるように並べる

### 原則

見出し、本文、補助情報、操作要素の間には、
**読み順・重要度・関係性** が分かるだけの視覚差を持たせる。

### 理由

ユーザーは 1 文字ずつ読んで理解するのではなく、
視覚的なまとまりと強弱から情報構造を推測する。
階層とグルーピングが弱いと、毎回どこから読むべきか迷う。

### 見落としサイン

- 全要素が同じ強さで並んでいる
- 関係する情報が離れている
- 無関係な要素が近すぎて一塊に見える

### 自己レビュー

- どこから読み始めるべきか自然に分かるか
- 関係するものが近く、無関係なものが離れているか
- 補助情報が主情報より目立っていないか

---

## P9. 画面密度はタスクに合わせて決める

### 原則

情報量は「少なければ良い」「多ければ強い」ではなく、
その画面で行うタスクの性質に合わせて調整する。

### 理由

ミニマルさは万能ではない。
一覧性や比較が重要な画面では一定の密度が必要であり、
逆に単純な完了フローでは情報を絞った方が良い。

### 見落としサイン

- 余白は多いが判断材料が足りない
- 情報は多いが読み分けられない
- 一覧画面なのに比較しづらい
- 単純作業なのに情報が多すぎる

### 自己レビュー

- この画面のタスクは「読む」「比較する」「入力する」「選ぶ」のどれが中心か
- そのタスクに対して情報量は多すぎないか / 少なすぎないか
- 密度を上げた結果、読み順や操作性を失っていないか

---

## P10. 熟練者の効率も考える

### 原則

デスクトップアプリでは、発見可能性だけでなく反復効率も重要である。
頻繁に行う操作には、ショートカットや素早い導線があるとよい。

### 理由

同じ操作を何度も行うユーザーにとって、
毎回同じクリック経路をたどる UI は摩擦が大きい。
特に業務アプリや制作ツールでは効率が UX の一部になる。

### 見落としサイン

- 頻出操作なのに毎回深い階層を辿る
- ショートカットがなく、マウス移動が多い
- 熟練者向けの近道が一切ない

### 自己レビュー

- よく使う操作を素早く実行できるか
- ショートカットがある場合、見つけられる形で示されているか
- 効率化のために初心者の理解可能性を壊していないか

---

## P11. UI はコンテンツや作業の邪魔をしない

### 原則

UI フレームや装飾は、作業対象より目立ちすぎないようにする。
ユーザーが見たいのは、通常、ツールそのものではなく、
ツールを使って扱うコンテンツや結果である。

### 理由

サイドバー、ツールバー、装飾、影、色などが過剰になると、
本来注目すべき作業対象から視線を奪う。

### 見落としサイン

- コンテンツより装飾の方が目立つ
- 常時表示のパネルが多く、主領域が狭い
- 「便利そう」な要素が増え続けている

### 自己レビュー

- 一番見てほしい内容が一番目立っているか
- 補助 UI が主作業領域を圧迫していないか
- 非常時以外は隠せる補助要素を常時出していないか

---

## P12. 美しさは独自性より整合性から生まれる

### 原則

見た目を整えるときは、派手な独自表現を足すより、
既存のトークン、余白、整列、書体、コントラスト、部品の整合を高める。

### 理由

多くの UI で品質感を下げるのは「個性のなさ」よりも、
**微妙なズレの蓄積** である。
統一感は、局所的な派手さより強く品質を支える。

### 見落としサイン

- 余白や角丸が場所ごとに微妙に違う
- 同種要素なのにサイズや書式が揃っていない
- 単発の例外スタイルが増えている

### 自己レビュー

- 今回の調整は全体の整合性を高めているか
- 既存トークンや共通部品で表現できないか
- 好みで足した差分になっていないか

# 4. 実装後セルフレビューの手順

UI を実装したあと、最低限次の順で見直す。

## 1. 目的確認

- この画面の主目的は一言で説明できるか
- その目的に対して UI が過不足なく支援しているか

## 2. 状態確認

- 処理中・成功・失敗・選択状態が分かるか

## 3. 立て直し確認

- ミスしたときに戻れるか、直せるか、やり直せるか

## 4. 認知負荷確認

- 最初の一手が分かるか
- 読み順と関係性が自然に見えるか

## 5. 一貫性確認

- 既存画面や既存部品と矛盾していないか
- OS の慣習から大きく外れていないか

## 6. アクセシビリティ確認

- キーボード操作、フォーカス可視性、ラベル、色依存の有無を確認したか

## 7. 効率確認

- 頻出操作を不必要に重くしていないか

## 8. 最後に視覚整理

- 余白、整列、コントラスト、ノイズを整理したか

# 5. 変更理由の書き方

実装や修正の説明では、次の形で理由を残す。

```md
### UI/UX Review Note

- Goal: この画面/変更で達成したいこと
- Principle: どの原則に基づく判断か
- Reason: なぜその変更が必要か
- Before: 変更前に何が分かりにくかったか / 使いづらかったか
- After: 変更後に何が分かりやすくなったか / 使いやすくなったか
- Trade-off: 何を残し、何を優先したか
```

### 例

```md
### UI/UX Review Note

- Goal: 送信画面で二重送信を減らす
- Principle: P1 状態は常に見えるようにする
- Reason: 押下後に無反応に見え、再クリックを誘発していたため
- Before: 送信中かどうか分からず不安になる
- After: 送信中表示とボタン状態変化で進行が分かる
- Trade-off: 軽微な処理でも状態表示が増えるが、誤認防止を優先した
```

# 6. 要約

この文書の目的は、エージェントに「勝手に止まれ」と命じることではない。
目的は、**実装した UI を自分で見直し、なぜその形がよいのかを説明しながら改善できるようにすること** である。

そのために、エージェントは次を意識する。

- まず操作できることを守る
- 次に状態と結果を見えるようにする
- 失敗しても立て直せるようにする
- 一貫性と OS 文脈を尊重する
- アクセシビリティを操作性の一部として扱う
- 最後に視覚的な品質を整える

そして各変更について、
**「どの原則に基づく判断か」「なぜそれが必要か」「何が良くなったか」** を説明できるようにする。

以上を、デスクトップ UI/UX の実装・自己レビュー原則とする。
