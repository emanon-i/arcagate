# Desktop UI Rules Part 2 — P4-P6 (consistency / OS / a11y)

[desktop-ui-rules.md](./desktop-ui-rules.md) の続編。

## P4. 同じ意味のものは同じように扱う

### 原則

同じ意味・同じ役割のものは、
画面が変わってもできるだけ同じラベル・同じ挙動・同じ見え方で扱う。

### 理由

一貫性は、ユーザーが「ここでもたぶん同じだろう」と予測する助けになる。
予測可能性が上がると、再学習のコストが減る。

### 見落としサイン

- 似た機能に別の名前が付いている
- 画面ごとに保存・閉じる・戻るの挙動が違う
- 類似コンポーネントが微妙に異なるスタイルで乱立している

### 自己レビュー

- この操作は、別の画面でも同じ意味で使われているか
- 既存の共通コンポーネントやトークンに寄せられるか
- 今回だけの都合で新しい表現を増やしていないか

### 補足: ラベルはアイコン名ではなく機能・状態・アクションを書く

ボタンやチップの **ラベル文字** は、隣のアイコンの形状名・記号名ではなく、
押したときに何が起きるか / 現在どの状態かを書く。

| 違反             | 正                  | 理由               |
| ---------------- | ------------------- | ------------------ |
| ⭐ + 「星」      | ⭐ + 「お気に入り」 | 「星」はアイコン名 |
| ☰ + 「三本線」   | ☰ + 「メニュー」    | 同上               |
| ＋ + 「プラス」  | ＋ + 「追加」       | 同上               |
| 🔍 + 「虫眼鏡」  | 🔍 + 「検索」       | 同上               |
| 🗑 + 「ゴミ箱」   | 🗑 + 「削除」        | 同上               |
| ⚙ + 「歯車」     | ⚙ + 「設定」        | 同上               |
| ✓ + 「チェック」 | ✓ + 「完了」        | 同上               |

書き分け:

- アクションボタン → **動詞**（追加 / 削除 / 編集 / 起動 / 検索）
- 状態切替 → **状態名**（お気に入り / 表示 / 非表示）
- ナビゲーション → **行き先名**（設定 / ライブラリ / ホーム）

`aria-label` も同じ。スクリーンリーダーで「星ボタン」と読まれても何の機能かわからない。

### 補足 2: 同じ機能には同じアイコン + 同じラベル（画面間の一貫性）

「Library 機能」を表すアイコンとラベルは、本体タブでも Settings サイドバーでも同じものを使う。
画面ごとに別アイコン・別ラベルを書くと、ユーザーは別機能だと誤認する。

実装ルール:

- 共有ナビ項目は `src/lib/nav-items.ts` で 1 箇所定義
- 各画面 (TitleBar / SettingsPanel / Sidebar 等) は import して参照
- 直接 lucide icon を import + label 文字列直書きは禁止（ナビ項目に限る）

### 補足 3: 横展開原則（不整合を見つけたら全画面チェック）

1 つの不整合を発見したら、同種パターンが他の画面にも存在する前提で全画面 audit:

- 全 `<button>` `<a>` `<TitleTab>` `<TitleAction>` `<MenuItem>` の icon + label が機能を説明しているか
- 同じ機能を扱う画面間で icon / label が揃っているか
- 言語混在（英 + 日）が起きていないか

1 ファイル fix で終わらせない。同じ間違いは必ず他にもある。

## P5. OS とアプリの文脈に逆らわない

### 原則

デスクトップアプリは単独で存在するのではなく、
Windows / macOS の文脈の中で使われる。
そのため、ショートカット、ウィンドウ挙動、ファイル操作、テキスト編集などは、
OS の期待にできるだけ沿う。

### 理由

デスクトップでは、ユーザーは OS 全体で身についた筋記憶や期待を持っている。
そこから外れると、違和感ではなく実際の操作ミスにつながる。

### 見落としサイン

- Ctrl / Cmd の扱いが曖昧
- テキスト選択や右クリックの挙動が独特
- ウィンドウ移動・リサイズ・保存ダイアログが期待と違う

### 自己レビュー

- この挙動は OS の標準期待と大きくずれていないか
- キーボードショートカットは Windows / macOS で自然か
- テキスト編集やファイル操作で変な癖を作っていないか

## P6. アクセシビリティは追加要件ではなく操作性そのものと考える

### 原則

アクセシビリティは「余裕があれば足すもの」ではなく、
その UI が本当に操作可能かどうかを決める中核要素として扱う。

### 理由

フォーカスが見えない、ラベルが失われている、色だけで状態を示す、といった問題は、
一部ユーザーだけの問題ではなく、UI の理解可能性と操作可能性そのものを下げる。

### 見落としサイン

- キーボードだけで主要操作に辿れない
- フォーカス位置が見えない
- アイコンだけで意味を伝えようとしている
- ライト/ダークの一方で読みにくい

### 自己レビュー

- Tab 移動だけでも主要フローを進められるか
- フォーカスがどこにあるか一目で分かるか
- 色が見分けにくくても状態を理解できるか
- ラベルや名前が削られていないか

## P7. フォームは「何をどう入れればよいか」が持続的に分かるようにする

### 原則

フォームでは、項目名、制約、単位、必須性、エラー箇所が
入力中も失われないようにする。

### 理由

フォームではユーザーは記憶ではなく画面上の手掛かりに頼って入力する。
項目名や制約が消えると、入力負荷とエラー率が上がる。

### 見落としサイン

- 入力を始めた途端に項目名が見えなくなる
- 送信するまで制約が分からない
- エラーがどの項目に対応するか分からない

### 自己レビュー

- 入力後も何の欄か分かるか
- 必須/任意や書式制約が必要な場所で見えているか
- エラー時にどこを直せばよいかすぐ分かるか

## P8. 情報は読み順と関係性が見えるように並べる

### 原則

見出し、本文、補助情報、操作要素の間には、
**読み順・重要度・関係性** が分かるだけの視覚差を持たせる。

### 理由

ユーザーは 1 文字ずつ読んで理解するのではなく、
視覚的なまとまりと強弱から情報構造を推測する。
階層とグルーピングが弱いと、毎回どこから読むべきか迷う。

### 見落としサイン

- 全要素が同じ強さで並んでいる
- 関係する情報が離れている
- 無関係な要素が近すぎて一塊に見える

### 自己レビュー

- どこから読み始めるべきか自然に分かるか
- 関係するものが近く、無関係なものが離れているか
- 補助情報が主情報より目立っていないか
