## 背景知識

このコマンドは以下の2つの知見に基づいています。

### 論文の知見（Gloaguen et al., 2026 "Evaluating AGENTS.md"）

AGENTS.md / CLAUDE.md の効果を4エージェント×2ベンチマークで実証評価した研究。主な発見:

- LLM生成のコンテキストファイルはタスク成功率を平均3%低下させ、推論コストを20%以上増加させる
- 原因は「コードから読み取れる情報の再記述」が大半を占めること。ディレクトリ構造のオーバービューはファイル到達速度を改善しない
- エージェントは指示に忠実に従う（ツール言及で使用率160倍増）ため、不要な指示もコスト増につながる
- 既存ドキュメントを全削除した環境ではコンテキストファイルが有効（+2.7%）。つまり「既存ドキュメントと重複しない情報」に価値がある
- 人間作成のコンテキストファイルはLLM生成を一貫して上回った。人間はエージェントが自力で知り得ない暗黙知を選んで書く傾向があるため

### Boris Cherny氏の知見（Claude Codeの生みの親がSNSで公開した知見を有志が構造化した行動規範）

主なポイント:

- Planモードを基本とし、うまくいかなくなったら即座に再計画する
- サブエージェントでコンテキストウィンドウを汚さない
- ユーザーからの修正は プロジェクトのMEMORY.md に記録し、同じミスを防ぐ
- 動作を証明するまで完了としない。「スタッフエンジニアが承認するか？」を基準にする
- バグレポートは質問せず自律的に修正する

## 生成モード

CLAUDE.md が存在しません。このプロジェクトのコードとコミット履歴を調査し、CLAUDE.md を新規作成してください。

### 行動規範（以下をそのまま記載する）

#### ワークフロー

- 3ステップ以上またはアーキテクチャに関わるタスクは必ずPlanモードで開始する
- 途中でうまくいかなくなったら、無理に進めずすぐに立ち止まって再計画する
- 構築だけでなく、検証ステップにもPlanモードを使う
- 実装前に詳細な仕様を書き、曖昧さを減らす

#### サブエージェント活用

- メインのコンテキストウィンドウをクリーンに保つためにサブエージェントを積極的に使う
- リサーチ・調査・並列分析はサブエージェントに任せる
- 1サブエージェント＝1タスクで集中させる

#### 自己改善

- ユーザーから修正を受けたら必ず `docs/lessons.md` にそのパターンを記録する
- 同じミスを繰り返さないよう自分へのルールを書き、ミス率が下がるまで改善し続ける
- セッション開始時に関連する lessons をレビューする

#### 完了基準

- 動作を証明できるまでタスクを完了としない
- テスト実行・ログ確認・差分確認で正しさを示す
- 「スタッフエンジニアはこれを承認するか？」と自問する

#### コア原則

- シンプル第一: 変更は最小限に。影響するコードを絞る
- 手を抜かない: 根本原因を見つける。一時的な修正は避ける
- 影響を最小化: 必要な箇所のみ変更し、バグを引き込まない
- エレガントさの追求: 重要な変更には「もっと良い方法はないか？」と立ち止まる。ただし単純な修正には過剰設計しない

#### 自律性

- バグレポートを受けたら質問せずにそのまま修正する
- ログ・エラー・失敗テストを自分で見て解決する
- 失敗しているCIテストは言われなくても修正しに行く

### プロジェクトの暗黙知（コードを調査して書く）

以下の観点でコードとコミット履歴を調査し、該当する情報を書いてください。該当しない項目は省略してください。

1. **設計方針・依存ルール**: レイヤー構成、依存方向、責務の境界
2. **禁止事項**: やってはいけないパターン
3. **意図的な設計判断とその理由**: 一見非効率・非標準に見える箇所の説明
4. **エラーハンドリング方針**: 例外/Result型/エラーコード等の統一方針
5. **開発ツールのコマンド**: パッケージ管理・テスト実行・lint のコマンド名だけ。使い方の詳細は不要
6. **CI とローカルの挙動差**: 差分がある場合のみ

### 書いてはいけないこと

- ディレクトリ構造の一覧や説明
- README や既存ドキュメントと重複する内容
- コーディングスタイル・フォーマット規則（linter設定に任せる）
- ライブラリの一般的な使い方
- 網羅的・長大なガイドライン

### 制約

- 暗黙知セクションは50行以内
- 各項目は1-2文で簡潔に
- 根拠がないことは推測で埋めない

## チェックモード

CLAUDE.md が既に存在します。内容を以下の観点でレビューし、問題点と改善案を報告してください。

### A. 含まれるべき内容が欠けていないか

以下は「コードを読むだけではエージェントが判断を誤るポイント」であり、明示が必要な項目です（論文の「エージェントが自力で知り得ない情報に価値がある」という原則から導出）。コードを実際に読んで、CLAUDE.md に書かれていない暗黙知があれば指摘してください。

1. **設計方針・依存ルール** — コードから現状のパターンは見えるが「今後もそうすべきか」の意思は見えないため
2. **禁止事項** — 頻度が低い例外的パターンは既存コードの傾向から学習しにくいため
3. **意図的な設計判断** — 理由がないとエージェントが善意でリファクタリングしてしまうため
4. **エラーハンドリング方針** — 統一方針がないとエージェントは場所ごとに異なるスタイルで書くため
5. **開発ツールのコマンド** — ツール名の指定はエージェントに確実に従われることが実証されているため（論文で確認済み）
6. **CI とローカルの挙動差** — CI設定ファイルだけからは読み取りにくい差分があるため

### B. 書くべきでない内容が含まれていないか

以下は実証研究でエージェントの性能低下やコスト増加が確認された項目です。

1. **ディレクトリ構造の列挙** — ls/find で取得可能。オーバービューはファイル到達速度を改善せず、不要な探索を誘発してステップ数・コスト増（論文で確認済み）
2. **既存ドキュメントとの重複** — エージェントは README を自力で読める。重複はコンテキストウィンドウを消費し性能低下（既存ドキュメントがある環境でLLM生成コンテキストファイルは成功率-3%、論文で確認済み）
3. **コーディングスタイル規則** — .editorconfig, ruff.toml 等で機械的に強制すべき。散文だと二重管理で矛盾リスク（論文の原則から導出: linter設定はエージェントが自力で読める情報）
4. **ライブラリの一般的な使い方** — LLMの事前知識に含まれる（論文の原則から導出: エージェントが既に知っている情報）
5. **網羅的・長大な記述** — 指示が増えるほど推論トークン14-22%増、ステップ数平均2.5-4増（論文で確認済み）。暗黙知セクションが50行を超えていたら警告

### C. 行動規範が含まれているか

以下はBoris Cherny氏（Claude Codeの生みの親）がSNSで公開した知見に基づく項目です。不足があれば追加を提案してください。

1. **Planモードの使用基準** — Claudeは一度走り出すと突き進む傾向があり、明示的なブレーキが必要
2. **サブエージェント活用方針** — コンテキストウィンドウの圧迫を防ぎ、並列処理で効率化するため
3. **自己改善ループ** — セッションをまたいで学習しないため、MEMORY.md で知識を永続化する必要がある
4. **完了基準** — 「動くことを証明するまで完了としない」を明示しないと自己申告完了になるため
5. **コア原則** — 過剰実装や応急処置を抑制するため
6. **自律性** — 「質問せず修正する」を明示しないとエージェントが逆質問で手を止めるため
